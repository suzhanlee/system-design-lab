# 검증 항목 상세 (Rubric)

> LLM이 객관적으로 PASS/FAIL/WARNING을 판정할 수 있는 루브릭 형식

---

## 1. Feasibility (구현 가능성)

### PASS 조건 (모두 충족 시)

- [ ] **기술 스택**: 요구사항이 Java 17+, Spring Boot 3.x 기술 범위 내
  - 예: Spring Data JPA, Spring Security, Spring MVC 사용
- [ ] **외부 의존성**: 모든 외부 API/라이브러리가 사용 가능 상태
  - 공개 API이거나 접근 권한이 명시됨
  - 라이선스 문제 없음 (MIT, Apache 2.0 등)
- [ ] **일정**: 핵심 기능이 요청 기간 내 구현 가능
  - 기간이 명시되지 않았다면 합리적인 기간 가정
- [ ] **리소스**: 필요한 인프라/인력이 프로젝트 범위 내
  - MySQL DB, 표준 인프라 구성

### FAIL 조건 (하나라도 해당 시)

- [ ] **기술 스택**: 프로젝트 기술 범위를 벗어남
  - 예: Python, Node.js, .NET 요구
  - 예: Spring Cloud, Kafka 등 미정의 기술 스택 요구
- [ ] **외부 의존성**: 접근 불가능한 API 또는 라이선스 문제
  - 폐쇄형 API (접근 권한 없음)
  - 상업적 라이선스 필요
- [ ] **일정**: 핵심 기능 구현 불가능한 기간
  - 예: 1주 내 대규모 시스템 구축
- [ ] **리소스**: 프로젝트 인프라 범위 초과
  - 예: Kubernetes, Redis Cluster 등

### WARNING (리스크 있음)

- [ ] **기술 스택**: 사용 가능하지만 경험 부족 가능
  - Spring WebFlux, R2DBC 등
- [ ] **외부 의존성**: 베타 API 또는 SLA 미보장
- [ ] **일정**: 버퍼 없는 타이트 일정 (±20% 여유 없음)
- [ ] **리소스**: 승인 필요한 인프라 리소스

## 2. Completeness (완전성)

### PASS 조건 (모두 충족 시)

- [ ] **기능 요구사항**: 모든 기능이 명세됨
  - 입력, 처리, 출력이 정의됨
  - CRUD 등 기본 동작이 명확함
- [ ] **예외 케이스**: 주요 에러 상황이 정의됨
  - 최소 3가지 이상의 예외 상황
  - 예: 중복 데이터, 권한 없음, 데이터 없음
- [ ] **비기능 요구사항**: 성능/보안/확장성 요구사항 존재
  - 응답 시간, 동시 사용자 수 등
  - 또는 "일반적인 수준" 명시
- [ ] **사용자 시나리오**: 주요 사용자 흐름이 정의됨
  - 정상 흐름 (Happy Path) 명시
  - 사용자 관점의 시나리오

### FAIL 조건 (하나라도 해당 시)

- [ ] **기능 요구사항**: 핵심 기능이 누락됨
  - "사용자 관리"만 있고 구체적 동작 없음
  - "자세한 내용은 추후" 등 회피성 표현
- [ ] **예외 케이스**: 예외 처리 전혀 없음
- [ ] **비기능 요구사항**: 완전히 누락
- [ ] **사용자 시나리오**: 사용자 흐름 정의 없음

### WARNING (보완 필요)

- [ ] **기능 요구사항**: 일부 세부사항 누락
  - 유효성 검증 규칙 미명시
  - 필수/선택 필드 구분 없음
- [ ] **예외 케이스**: 1~2개만 정의됨 (최소 3개 권장)
- [ ] **비기능 요구사항**: 모호한 표현만 존재
  - "빠른 응답", "높은 가용성" 등 수치 없음
- [ ] **사용자 시나리오**: 일부 분기만 정의됨

## 3. Consistency (일관성)

### PASS 조건 (모두 충족 시)

- [ ] **충돌**: 요구사항 간 모순이 없음
  - 동일 기능에 상반된 요구 없음
- [ ] **모호성**: 구체적인 수치/기준 사용
  - "3초 이내", "1000 TPS", "99.9% 가용성"
- [ ] **용어**: 동일 개념에 동일 용어 사용
  - 용어 사전 또는 일관된 명명 규칙
- [ ] **수준**: 요구사항의 상세도가 일관적
  - 모든 기능이 유사한 수준의 상세도

### FAIL 조건 (하나라도 해당 시)

- [ ] **충돌**: 명백한 모순 존재
  - 예: "모든 사용자 접근 허용" vs "관리자만 접근"
- [ ] **모호성**: 핵심 기능에 모호한 표현 다수
  - "적절한", "필요한", "충분한" 등 판단 기준 없음
- [ ] **용어**: 동일 개념에 여러 용어 혼용
  - "사용자", "유저", "회원" 등 무분별 혼용
- [ ] **수준**: 상세도 편차 심함
  - 일부는 구체적, 일부는 추상적

### WARNING (경미한 문제)

- [ ] **충돌**: 잠재적 충돌 가능성 (해석에 따라 다름)
- [ ] **모호성**: 일부 항목에 모호한 표현 존재
- [ ] **용어**: 1~2개의 용어 혼용 (문맥으로 구분 가능)
- [ ] **수준**: 일부 항목만 상세도 차이

## 4. Dependencies (의존성)

### PASS 조건 (모두 충족 시)

- [ ] **외부 API**: 연동 API 목록이 명확히 정의됨
  - API명, 목적, 필수/선택 여부
  - 또는 "외부 API 없음" 명시
- [ ] **데이터베이스**: 필요 DB/스키마가 정의됨
  - 주요 엔티티, 데이터 유형
  - 또는 프로젝트 기본 DB 사용 명시
- [ ] **타 시스템**: 연동 필요 내/외부 시스템 명시
  - 시스템명, 연동 목적, 데이터 흐름
- [ ] **인프라**: 필요 인프라 리소스 정의됨
  - 또는 "기본 인프라 사용" 명시

### FAIL 조건 (하나라도 해당 시)

- [ ] **외부 API**: API 연동 필요하지만 명세 없음
  - "외부 시스템 연동"만 언급, 구체적 API 없음
- [ ] **데이터베이스**: 데이터 저장 필요하지만 스키마 정의 없음
- [ ] **타 시스템**: 시스템 연동 언급되었으나 상세 없음
- [ ] **인프라**: 특수 인프라 필요하나 정의 없음

### WARNING (보완 필요)

- [ ] **외부 API**: API 목록은 있으나 상세 스펙 누락
  - 엔드포인트, 인증 방식 미명시
- [ ] **데이터베이스**: 주요 엔티티만 정의, 상세 필드 누락
- [ ] **타 시스템**: 시스템명만 있고 데이터 흐름 미명시
- [ ] **인프라**: 일부 리소스만 정의됨

## 5. Epic 전용 검증 항목 (Epic 모드 시에만 적용)

### PASS 조건 (모두 충족 시)

- [ ] **의존 Epic**: 의존하는 Epic이 먼저 완료/검증됨
  - 의존 Epic 없음 또는 완료된 Epic만 의존
  - 순환 의존성 없음
- [ ] **범위**: 포함/제외 기능이 명확함
  - In-scope: [구체적 기능 목록]
  - Out-of-scope: [제외 기능 및 사유]
- [ ] **DoD (Definition of Done)**: 완료 기준이 테스트 가능함
  - 검증 가능한 조건 (예: "모든 테스트 통과")
  - 단순 "완료"가 아닌 구체적 기준
- [ ] **Entity**: Entity 설계가 적절함 (1~2개 권장)
  - 단일 Epic 당 1~2개 Entity
  - Entity 간 관계가 명확함

### FAIL 조건 (하나라도 해당 시)

- [ ] **의존 Epic**: 미완료 Epic에 의존
  - 아직 구현되지 않은 Epic을 의존
- [ ] **범위**: 포함/제외 기능이 불명확
  - "기본 기능", "필요한 기능" 등 모호함
- [ ] **DoD**: 완료 기준이 없거나 테스트 불가능
  - "완료", "개발 완료" 등 추상적 표현만
- [ ] **Entity**: Entity 설계 부적절
  - 3개 이상 Entity (Epic 범위 과대)
  - Entity 정의 없음

### WARNING (주의 필요)

- [ ] **의존 Epic**: 의존 Epic이 진행 중
- [ ] **범위**: 일부 경계가 모호함
- [ ] **DoD**: 일부 기준만 테스트 가능
- [ ] **Entity**: 2개 Entity (경계선)

## 판정 가이드

### 최종 판정 규칙

| 조건 | 판정 |
|------|------|
| FAIL 조건 1개 이상 해당 | **FAIL** |
| WARNING 3개 이상 해당 | **WARNING** |
| WARNING 1~2개 + PASS 나머지 | **PASS** (주석 필요) |
| 모든 PASS 조건 충족 | **PASS** |

### 판정 출력 형식

```markdown
## [항목명] 검증 결과: PASS | WARNING | FAIL

### 충족 조건
- [x] 만족한 조건
- [x] 만족한 조건

### 미충족/문제 조건
- [ ] 문제 조건 (구체적 사유)

### 판정 사유
[LLM이 판정한 이유를 1~2문장으로 설명]
```
