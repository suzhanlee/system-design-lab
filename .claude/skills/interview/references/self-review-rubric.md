# Self-Review 채점 기준

## 목적
Phase D에서 사용자가 작성한 요구사항 초안을 스스로 평가하여 품질 검증한다.

---

## Self-Review 체크리스트

### 평가 항목 (각 1~5점)

| # | 질문 | 점수(1~5) |
|---|------|-----------|
| 1 | **완전성**: 모든 Must Have 기능이 포함되었는가? | |
| 2 | **구체성**: 각 요구사항이 측정 가능한가? (수치/조건 포함) | |
| 3 | **일관성**: 요구사항 간 충돌이 없는가? | |
| 4 | **실현 가능성**: 기술적/일정적으로 구현 가능한가? | |
| 5 | **명확성**: 6개월 후에도 누구나 이해할 수 있는가? | |

---

## 평가 기준

### 점수 가이드

| 점수 | 의미 | 기준 |
|------|------|------|
| 5 | 완벽 | 의문점 전혀 없음, 즉시 구현 가능 |
| 4 | 양호 | 소소한 모호함 있으나 큰 문제 없음 |
| 3 | 보통 | 일부 명확화 필요, 구현에 지장 있을 수 있음 |
| 2 | 미흡 | 주요 항목 누락/모호, 재작성 권장 |
| 1 | 부족 | 근본적으로 다시 작성 필요 |

---

## 등급 판정

| 총점 | 등급 | 조치 |
|------|------|------|
| 22-25 | **A** | 바로 `/validate` 진행 가능 |
| 18-21 | **B** | 소수 항목 보완 권장 |
| 14-17 | **C** | 여러 항목 보완 후 `/validate` |
| 10-13 | **D** | 전면 수정 권장 |
| 5-9 | **F** | 처음부터 다시 작성 |

---

## Self-Review 예시

```markdown
# Self-Review 결과

## 점수

| # | 질문 | 점수 | 비고 |
|---|------|------|------|
| 1 | 완전성 | 4 | Must Have 5개 모두 포함 |
| 2 | 구체성 | 3 | "빠른 응답"을 수치화 필요 |
| 3 | 일관성 | 5 | 충돌 없음 |
| 4 | 실현 가능성 | 4 | 기술 스택 호환 |
| 5 | 명확성 | 4 | 비개발자도 이해 가능 |

**총점**: 20/25 (등급 B)

## 개선 사항
- "빠른 응답" → "P95 응답 시간 500ms 이내"로 구체화 필요
```

## Self-Review 수행 가이드

### 1. 6개월 테스트
```
6개월 후 내가 이 문서를 다시 읽었을 때:
- 어떤 기능을 만들어야 하는지 명확한가?
- 기술적 세부사항이 충분한가?
- 누락된 정보가 없는가?
```

### 2. 개발자 시뮬레이션
```
이 요구사항만 보고 개발자가 바로 코딩을 시작할 수 있는가?
- 아니라면 무엇이 부족한가?
```

### 3. 반대 질문
```
- "이 요구사항이 실패하려면?" → 누락된 예외 케이스 발견
- "경쟁 서비스는 어떻게 하나?" → 기준점 확인
```

## STOP Protocol (Phase D 완료)

Self-Review 완료 후:

```
---
✅ Self-Review 완료

총점: [X]/25 (등급: [A/B/C/D/F])

[등급이 B 이상인 경우]
다음 단계: /validate

[등급이 C 이하인 경우]
개선 후 다시 Self-Review를 진행하세요.
```
