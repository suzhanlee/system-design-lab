# Red Team 6가지 검토 관점 체크리스트

---

## 1. Security (보안)

### 인증/인가
- [ ] 인증 메커니즘이 안전한가?
- [ ] 권한 체크가 모든 접근 지점에서 수행되는가?
- [ ] 세션 관리가 안전한가?

### 데이터 보호
- [ ] 민감 데이터가 암호화되는가?
- [ ] 암호화 알고리즘이 적절한가?
- [ ] 키 관리가 안전한가?

### 입력 검증
- [ ] 모든 사용자 입력이 검증되는가?
- [ ] SQL Injection 방지 조치가 있는가?
- [ ] XSS 방지 조치가 있는가?
- [ ] CSRF 방지 조치가 있는가?

### 외부 연동
- [ ] 외부 API 호출이 안전한가?
- [ ] 타임아웃이 적절히 설정되었는가?
- [ ] 에러 처리가 정보 노출을 방지하는가?

### 로깅/감사
- [ ] 보안 이벤트가 로깅되는가?
- [ ] 민감 정보가 로그에 포함되지 않는가?

---

## 2. Performance (성능)

### 데이터베이스
- [ ] N+1 Query 문제가 없는가?
- [ ] 적절한 인덱스가 정의되었는가?
- [ ] 대량 데이터 처리 방안이 있는가?
- [ ] 커넥션 풀 크기가 적절한가?

### 캐싱
- [ ] 캐싱 전략이 정의되었는가?
- [ ] 캐시 무효화 전략이 있는가?
- [ ] 핫 데이터와 콜드 데이터가 구분되는가?

### 네트워크
- [ ] API 응답 크기가 적절한가?
- [ ] 불필요한 네트워크 호출이 없는가?
- [ ] 압축이 사용되는가?

### 연산
- [ ] 무거운 연산이 비동기로 처리되는가?
- [ ] 병목 지점이 식별되었는가?
- [ ] 응답 시간 목표가 달성 가능한가?

---

## 3. Scalability (확장성)

### 수평 확장
- [ ] Stateful/Stateless 구분이 명확한가?
- [ ] 세션 저장소가 분리되었는가?
- [ ] 로드 밸런싱이 가능한 구조인가?

### 데이터 확장
- [ ] 데이터 증가 시 대응 방안이 있는가?
- [ ] 샤딩/파티셔닝 전략이 있는가?
- [ ] 아카이빙 전략이 있는가?

### 트래픽 확장
- [ ] 트래픽 스파이크 대응이 가능한가?
- [ ] Rate Limiting이 고려되었는가?
- [ ] Circuit Breaker가 적용되었는가?

### 비용
- [ ] 확장 시 비용 증가가 예측 가능한가?
- [ ] 비용 효율적인 확장 전략인가?

---

## 4. Maintainability (유지보수성)

### 코드 구조
- [ ] 모듈화가 잘 되어 있는가?
- [ ] 의존성 방향이 올바른가?
- [ ] 순환 의존성이 없는가?

### 문서화
- [ ] API 문서가 자동 생성되는가?
- [ ] 아키텍처 다이어그램이 있는가?
- [ ] 운영 가이드가 있는가?

### 테스트
- [ ] 테스트 커버리지가 충분한가?
- [ ] 테스트가 유지보수 가능한가?
- [ ] CI/CD가 구축되었는가?

### 기술 부채
- [ ] 하드코딩이 없는가?
- [ ] deprecated 기술이 사용되지 않는가?
- [ ] 새 팀원이 이해할 수 있는 구조인가?

---

## 5. Business (비즈니스)

### 요구사항 충족
- [ ] 모든 Must Have 요구사항이 반영되었는가?
- [ ] 비즈니스 규칙이 정확히 구현되었는가?
- [ ] 에지 케이스가 처리되었는가?

### 사용자 경험
- [ ] 에러 메시지가 사용자 친화적인가?
- [ ] 응답 시간이 사용자 기대를 충족하는가?
- [ ] 동시 사용자 시나리오가 고려되었는가?

### 규정/컴플라이언스
- [ ] 개인정보보호 규정을 준수하는가?
- [ ] 감사 요구사항이 충족되는가?
- [ ] 데이터 보관 정책이 정의되었는가?

### 확장성 (비즈니스)
- [ ] 새로운 비즈니스 요구사항 수용이 쉬운가?
- [ ] A/B 테스트가 가능한 구조인가?
- [ ] Feature Flag 사용이 고려되었는가?

---

## 6. Reliability (신뢰성)

### 장애 처리
- [ ] 장애 감지 메커니즘이 있는가?
- [ ] Failover 전략이 정의되었는가?
- [ ] 장애 전파 방지가 되는가?

### 복구
- [ ] 백업 전략이 있는가?
- [ ] 복구 절차가 문서화되었는가?
- [ ] RTO/RPO가 정의되었는가?

### 모니터링
- [ ] Health Check가 구현되었는가?
- [ ] 알림 시스템이 있는가?
- [ ] 메트릭 수집이 되는가?

### 데이터 무결성
- [ ] 트랜잭션 경계가 명확한가?
- [ ] 데이터 정합성이 보장되는가?
- [ ] 동시성 제어가 되는가?

---

## 심각도 분류 기준

| 심각도 | 기준 | 예시 |
|--------|------|------|
| **HIGH** | 즉시 조치 필요, 보안/데이터 손실 위험 | SQL Injection, 데이터 무결성 위반 |
| **MEDIUM** | 단기 내 조치 권장, 성능/확장성 문제 | N+1 Query, 캐시 미사용 |
| **LOW** | 장기 개선 사항, 유지보수성 이슈 | 문서화 부족, 하드코딩 |

---

## 검토 체크리스트 사용법

1. **ADR 읽기**: 검토할 ADR을 읽는다
2. **각 관점 스캔**: 6가지 관점에서 체크리스트 항목 확인
3. **이슈 식별**: 해당되는 항목에 대해 이슈 작성
4. **심각도 평가**: HIGH/MEDIUM/LOW 분류
5. **제안 작성**: 개선 방안 제시
6. **Critique Report 생성**: `.atdd/design/redteam/critique-[번호].md`
